# CLAUDE WORKFLOW RULES

## GENERAL BEHAVIOR

Think carefully before making changes.

Always:
- Analyze first
- Plan before coding
- Explain major decisions
- Maintain consistency
- Respect existing architecture

Do not:
- Rewrite unrelated files
- Break existing components
- Change architecture unnecessarily
- Generate low-quality placeholder UI

---

# WORKFLOW RULES

Follow this workflow strictly:

1. Analyze requirements
2. Plan architecture
3. Build folder structure
4. Create reusable UI system
5. Build components section-by-section
6. Optimize responsiveness
7. Improve animations
8. Refactor and optimize
9. Final polish

Never build the entire application at once.

---

# UI GENERATION RULES

All UI must:
- Look premium
- Feel modern
- Be responsive
- Maintain consistency
- Follow the design system
- Use proper spacing
- Have strong visual hierarchy

Avoid:
- Generic layouts
- Cluttered sections
- Weak typography
- Inconsistent spacing
- Cheap-looking effects

---

# CODING RULES

Always:
- Use TypeScript properly
- Reuse components
- Keep files organized
- Keep components small
- Write maintainable code
- Use clean Tailwind structure

Avoid:
- Massive components
- Duplicate code
- Overengineering
- Unused imports
- Unnecessary dependencies

---

# ANIMATION RULES

Use Framer Motion subtly.

Animations should feel:
- Smooth
- Intentional
- Premium
- Lightweight

Never:
- Hurt performance
- Overanimate
- Distract from content

---

# RESPONSIVENESS RULES

Before finishing any section:
- Test mobile layout mentally
- Ensure spacing consistency
- Ensure readable typography
- Ensure touch-friendly interactions

Mobile experience is HIGH priority.

---

# PROJECT RULES

Treat projects as:
- Real startup products
- Professional SaaS applications
- Production-ready systems

Each project showcase should:
- Explain the problem
- Explain the solution
- Showcase the UI
- Highlight the architecture
- Emphasize technical decisions

---

# EDITING RULES

Before editing:
- Understand current code structure
- Preserve working functionality
- Avoid unnecessary rewrites

When updating files:
- Modify only necessary parts
- Keep consistency
- Preserve code quality

---

# FINAL QUALITY STANDARD

Every generated section should feel:
- Production-ready
- Recruiter-impressive
- Modern
- Elegant
- Technically polished

The portfolio should NOT feel like:
- A beginner project
- A template
- A school requirement

It should feel like:
“A modern engineer building real products.”
